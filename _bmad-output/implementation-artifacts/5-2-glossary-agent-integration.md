## Story

As a **user starting work on a new project**,
I want **an AI-powered glossary agent that extracts and maintains a glossary of project-specific terms, abbreviations, stakeholder names, and domain jargon from my project documents**,
So that **I (and the @nest agent) can quickly understand project-specific language without manually compiling a dictionary**.

## Business Context

When onboarding to a new project, one of the first things users do after running `nest sync` is manually ask the `@nest` agent to generate a glossary file. This happens on **every single project** — it's a universal pattern that should be automated.

The glossary serves multiple purposes:
1. **Human onboarding:** New team members can quickly learn project-specific jargon, abbreviations, and who the key stakeholders are.
2. **Agent augmentation:** The `@nest` agent can reference `glossary.md` for better answers — understanding that "PDC" means "Project Delivery Committee" in this client's world, not a generic tech term.
3. **Institutional knowledge:** Terms accumulate over time as new documents are added, building a living dictionary.

This follows the same **two-phase pattern** established by Story 5.1 (Index Enrichment):
1. **Phase 1 (Python, deterministic):** During `nest sync`, extract candidate terms (abbreviations, proper nouns, repeated domain words) and write them to `00_GLOSSARY_HINTS.yaml`. No LLM calls in the CLI — keeps the tool privacy-first and avoids API key requirements.
2. **Phase 2 (VS Code Agent, LLM):** A shipped `nest-glossary.agent.md` template reads the hints + context files, reasons about definitions, and writes/updates `glossary.md`.

**Key design principle:** Human curation is sacred. Once a user refines a definition in `glossary.md`, the system never overwrites it. The glossary agent is additive-only for existing terms.

**And** the hints file is excluded from the master index

### AC2: Candidate Term Extraction Logic

**Given** a context file is scanned for candidate terms
**When** content analysis runs
**Then** the following patterns are detected:
- **Abbreviations:** Uppercase sequences of 2+ characters (e.g., `PDC`, `SOW`, `SME`) that appear at least 2 times across the project
- **Proper nouns:** Capitalized multi-word sequences that appear in stakeholder/people contexts (e.g., names followed by titles/roles)
- **Repeated domain terms:** Capitalized words or phrases that appear 3+ times and are not common English words

**And** common generic terms are filtered out (e.g., `API`, `URL`, `PDF`, `SQL`, `HTML`, `CSS`, `JSON`, `YAML`, `HTTP`, `REST`, `CLI`, `IDE`)
**And** each candidate includes up to 3 context snippets (surrounding sentence, truncated to 100 chars)
**And** source file paths are tracked per term

### AC3: Incremental Hints — Content Hash Tracking

**Given** glossary hints have been generated in a previous sync
**When** a subsequent sync runs and a file's content has not changed (checksum match in manifest)
**Then** the previously extracted terms from that file are reused without re-scanning
**And** only new or modified files are scanned for candidate terms

**Given** a file is deleted or orphaned
**When** the hints are regenerated
**Then** terms from that file are removed from the hints (but NOT from `glossary.md` — terms are never deleted from the curated glossary)

### AC4: Glossary Agent Template Shipped

**Given** `nest init` creates the project
**When** agent files are generated
**Then** `.github/agents/nest-glossary.agent.md` is created alongside `nest.agent.md`

### Critical Implementation Details

**Abbreviation Detection:**
```python
import re

ABBREVIATION_PATTERN = re.compile(r'\b[A-Z]{2,}\b')

def _find_abbreviations(content: str) -> dict[str, list[str]]:
    """Find uppercase abbreviations with context snippets."""
    results: dict[str, list[str]] = {}
    for match in ABBREVIATION_PATTERN.finditer(content):
        term = match.group()
        if term not in GENERIC_TERM_FILTER:
            # Extract surrounding sentence as context
            start = max(0, content.rfind('.', 0, match.start()) + 1)
            end = content.find('.', match.end())
            if end == -1:
                end = min(len(content), match.end() + 50)
            snippet = content[start:end].strip()[:100]
            results.setdefault(term, []).append(snippet)
    return results
```

**Proper Noun Detection:**
```python
# Heuristic: Capitalized multi-word sequences not at sentence start
PROPER_NOUN_PATTERN = re.compile(
    r'(?<=[a-z.,:;]\s)([A-Z][a-z]+(?:\s[A-Z][a-z]+)+)'
)
```
This is intentionally conservative — the LLM agent can do deeper analysis. The Python layer just finds *candidates*.

**Generic Term Filter:**
Maintained as a constant set in the service. Covers common tech abbreviations that appear in every project and provide no project-specific value.

**Incremental Processing:**
```python
def extract_all(
    self,
    context_dir: Path,
    changed_files: set[str] | None = None,
) -> GlossaryHints:
    """Extract candidate terms, only scanning changed files.

    Args:
        context_dir: Path to _nest_context/
        changed_files: Set of relative paths that changed in this sync.
                       If None, scan all files (first run).
    """
```

**Hints Merge Logic:**
```
Previous Hints + New Extractions → Merged Hints
  - For unchanged files: carry forward previous terms
  - For changed files: replace with new extractions
  - For deleted files: remove terms ONLY from hints (never from glossary.md)
  - Cross-file aggregation: merge occurrences and sources for same term
```

### Glossary Agent — Key Design Decisions

- **Project-specificity is key:** The agent must distinguish between generic terms (API, REST) and project-specific ones (PDC = Project Delivery Committee at *this* client)
- **Additive-only for existing terms:** The agent NEVER overwrites a human-edited definition. It can add new terms, but existing rows are untouched.
- **Category taxonomy:** Abbreviation | Stakeholder | Domain Term | Project Name | Tool/System | Other
- **Sentinel markers:** `<!-- nest:glossary-start -->` / `<!-- nest:glossary-end -->` enable reliable table parsing, consistent with the index table pattern
- **Sorting:** Alphabetical by Term — deterministic, easy to find terms
- **Context snippets in hints:** Give the agent enough context to write meaningful definitions without reading every source file

#### [LOW] L1: Incremental change detection uses metadata hash as proxy (NOT FIXED — by design)
Glossary change detection relies on metadata content hashes (headings + first_paragraph + lines) rather than full file checksums. A user-curated file could add abbreviations mid-document without changing structure. Documented as intentional trade-off in dev notes — metadata hash is a pragmatic approximation that avoids maintaining a separate content-hash tracking system for glossary extraction.
