Status: done

## Story

As a **user who has synced project documents**,
I want **the master index to contain a short description per file**,
so that **the @nest agent (and I) can quickly identify which file to read without opening every document**.

## Business Context

The current `00_MASTER_INDEX.md` lists filenames and nothing else. For projects with 50+ documents, filenames alone are insufficient for an LLM agent (or human) to locate relevant content. Adding short descriptions (up to 15 words) to each file entry transforms the index from a directory listing into a searchable catalogue.

Since Python cannot reason about document content, this requires a two-phase approach:
1. **Phase 1 (deterministic, Python):** Extract structural metadata (headings, first paragraph, line count, table columns) from each file and write it to a hints file.
2. **Phase 2 (LLM, agent):** An enricher agent reads the hints and writes short descriptions into the index.

This story covers **Phase 1 only** — all deterministic Python work: metadata extraction, index format change to a table with a Description column, incremental hint tracking via content hashes, and the sync-time prompt that tells the user to invoke the enricher agent.

The enricher agent template (Phase 2) is also delivered in this story as a shipped Jinja2 template, but the agent invocation itself is a user action in VS Code chat — not a Python subprocess.

### Critical Implementation Details

**Content Hash Algorithm:**
```python
import hashlib
import json

def _compute_content_hash(headings: list[HeadingInfo], first_paragraph: str, lines: int) -> str:
    """Deterministic hash of extracted metadata for change detection."""
    payload = json.dumps({
        "headings": [{"level": h.level, "text": h.text} for h in headings],
        "first_paragraph": first_paragraph,
        "lines": lines,
    }, sort_keys=True)
    return hashlib.sha256(payload.encode()).hexdigest()[:16]  # 16-char hex prefix
```
- Use `sort_keys=True` in JSON serialization for determinism
- Truncate SHA-256 to 16 hex chars — collision-resistant enough for this use case, keeps YAML readable

**Heading Extraction (Markdown only):**
```python
import re

HEADING_PATTERN = re.compile(r"^(#{1,6})\s+(.+)$", re.MULTILINE)

def _extract_headings(content: str) -> list[HeadingInfo]:
    return [
        HeadingInfo(level=len(match.group(1)), text=match.group(2).strip())
        for match in HEADING_PATTERN.finditer(content)
    ]
```

**First Paragraph Extraction:**
- Skip lines starting with `#` (headings)
- Skip empty lines
- Take first non-empty, non-heading line(s) up to 200 characters
- For non-Markdown files: take the first non-empty line, truncated to 200 chars
- For 0-byte files: return empty string

**Line Count — Cross-Platform:**
- Use `len(content.splitlines())` (not `content.count('\n')`) to handle `\n`, `\r\n`, and `\r` consistently across platforms
- 0-byte files → `lines=0`

**Index Table Parsing:**
```python
def parse_index_descriptions(content: str) -> dict[str, str]:
    """Extract file→description mapping from existing index table."""
    start = content.find(INDEX_TABLE_START)
    end = content.find(INDEX_TABLE_END)
    if start == -1 or end == -1:
        return {}
    table_block = content[start:end]
    descriptions: dict[str, str] = {}
    for line in table_block.splitlines():
        if line.startswith("|") and not line.startswith("| File") and not line.startswith("|--"):
            parts = [p.strip() for p in line.split("|")]
            # parts: ['', 'file.md', '123', 'description text', '']
            if len(parts) >= 5:
                file_path = parts[1]
                # Handle descriptions containing pipe characters:
                # Join parts[3:-1] to reconstruct the full description
                description = " | ".join(parts[3:-1]).strip()
                if file_path and file_path != "File":
                    descriptions[file_path] = description
    return descriptions
```

### Enricher Agent — Key Design Decisions

- **Ceiling, not range:** Instructions say "at most 15 words" rather than "10-12 words" — LLMs are bad at precise word counting, a ceiling works better
- **One-shot processing:** Agent fills all empty descriptions in a single pass — no looping or multi-turn
- **Idempotent:** Can be re-run safely; only touches empty Description cells
- **Token-efficient:** ~100 tokens input per file (from hints), ~15 tokens output per file
